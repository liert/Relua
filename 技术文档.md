# Lua 字节码（.luac）反编译从 0 到 1 的技术文档

> 目标：从 `.luac`（Lua 虚拟机字节码）文件反编译回**可读、可编译并尽量语义等价**的 Lua 源代码。

本文件面向工程实现者，列出实现该反编译器所需的全部技术、开源库、实现流程、关键数据结构、算法伪代码、测试与验证方法，以及常见混淆/防护的处理策略。文档按用户要求组织：环境搭建、字节码解析、控制流重建、代码生成、高级技术、验证测试与性能优化。

## 2. 字节码解析核心技术

### 2.1 文件头解析与验证技术

Lua 字节码文件头（header）包含魔数、版本号、端序、整数/浮点大小、lua_Number 格式等，是解析前必须验证的。

**关键步骤**：
1. 读取前 N 字节（通常 12~16 字节）并对照目标 Lua 版本的头结构。
2. 验证魔数（通常 `0x1B4C7561` 即 `\x1bLua`）与版本字段；若不匹配，尝试其他版本或字节序。
3. 解析后续标志：`endianness`、`size_int`、`size_size_t`、`size_Instruction`、`size_lua_Number`、`integral_flag`。
4. 使用头信息调整后续解析器（大端/小端，整数/浮点解码）。

**注意**：有些混淆器会改写头部或对字节序做异常处理；建议实现「容错模式」：尝试常见组合（小端/大端、32/64-bit）并做验证（看能否解析出合理的常量/字符串表）。

**头结构示例（伪C结构，Lua 5.1）**:
```c
struct LuaHeader {
  uint8_t signature[4]; // 0x1b 'L' 'u' 'a'
  uint8_t version;      // e.g. 0x51 for 5.1
  uint8_t format;       // usually 0
  uint8_t endianness;   // 1 = little, 0 = big
  uint8_t size_int;
  uint8_t size_size_t;
  uint8_t size_Instruction;
  uint8_t size_lua_Number;
  uint8_t integral_flag; // 0 = float, 1 = integral
};
```

### 2.2 函数原型（Proto）结构解析方法

Lua 字节码将函数（包括顶层 chunk）以 Proto（或称 `FunctionProto`）结构递归序列化。解析需要递归地重建每个 Proto 的字段。

**Proto 常见字段**：
- `source`（字符串）
- `line_defined`, `last_line_defined`（行信息）
- `num_upvalues`, `num_params`, `is_vararg`, `max_stack_size`
- `code`（指令数组）
- `constants`（常量表：nil / boolean / number / string 等）
- `protos`（嵌套 Proto 数组）
- `lineinfo`（指令到源行映射）
- `locvars`（局部变量表）
- `upvalues` 信息

**解析策略**：
- 以字节结构解析器读取字符串、整数、size_t、Instruction（通常为 4 字节）等。
- 递归 parseProto()：读取基本属性 -> 读取 code[] -> constants[] -> protos[] -> debug（lineinfo, locvars, upvalues）。
- 在解析过程中将数据填入内存表示（如 `Proto` 类）。

**伪码**：
```
function parseProto(reader):
  p = new Proto()
  p.source = reader.readString()
  p.line_defined = reader.readInt()
  p.last_line_defined = reader.readInt()
  p.num_upvalues = reader.readByte()
  p.num_params = reader.readByte()
  p.is_vararg = reader.readByte()
  p.max_stack_size = reader.readByte()
  p.code = reader.readInstructionArray()
  p.constants = reader.readConstants()
  p.protos = []
  for i in 1..reader.readInt():
    p.protos.append(parseProto(reader))
  p.lineinfo = reader.readLineInfo()
  p.locvars = reader.readLocVars()
  p.upvalues = reader.readUpvalues()
  return p
end
```

### 2.3 指令序列解析与转换技术

**Instruction 编码**：Lua 指令通常为固定大小（如 32 位），根据版本分为几种编码格式（例如 6 位 opcode + A/B/C 或 Bx/Ax）。应实现统一的 `Instruction` 解析器来拆分 opcode 与操作数字段。

**步骤**：
1. 从 `code[]` 读取原始指令整数。
2. 使用位域掩码（mask）和移位操作解码为 `(opcode, A, B, C, Bx, sBx)`。
3. 将低级指令转换为一个更高层次的 `Op` 对象（保存原始参数，便于后续 IR 转换）。

**Instruction 解码示例伪码**：
```
function decodeInstruction(ins):
  opcode = ins & 0x3F
  A = (ins >> 6) & 0xFF
  C = (ins >> 14) & 0x1FF
  B = (ins >> 23) & 0x1FF
  Bx = (ins >> 14) & 0x3FFFF
  sBx = Bx - bias
  return (opcode, A, B, C, Bx, sBx)
end
```

> 注意：`bias` 值依赖于指令字段宽度（通常 `2^{(n-1)}-1`）。确认目标版本字段长度。

### 2.4 常量池（constants）解析与重建策略

**常量类型**：`NIL, BOOLEAN, NUMBER, STRING`（不同版本还可能包含整数/浮点区分）。

**解析**：逐项读取常量表并保持原始类型与字节表示。字符串应处理 `` 与非 UTF-8 字符，保留原始字节以保证重写时字节等价。

**重建策略**：
- 对于数值常量，保留原始表示（浮点精度）；生成表达式时尽量把数字以原始格式输出（例如 `1.0` 而非 `1`，若原字节为浮点）。
- 对于重复字符串，保持单一常量池引用，代码生成时可按语义将短字符串转换为更可读形式。
- 对于表/复杂常量（有些版本支持内联表），实现表重建映射。

### 2.5 局部变量表与上值（upvalues）处理技术

**locvars**：包含变量名与有效指令区间 `[startpc, endpc)`。用于将寄存器/栈槽映射回源码变量名与作用域。

**upvalues**：有两种语义：外部局部变量或伪寄存器。解析 `upvalues` 表将帮助恢复闭包语义与外部引用。

**处理要点**：
- 根据 `locvars` 的生存期（startpc/endpc）来决定何时在生成的代码中插入局部变量声明。
- 当 `locvars` 缺失或被混淆，需基于访问模式（寄存器读写、函数调用）进行变量名推断与重命名（如 `v1, v2` → 更语义化名称）。

## 3. 控制流分析与重建

字节码为栈机/寄存器机器指令序列，源代码的结构化控制流需要通过静态分析从指令序列中识别出来。

### 3.1 基本块（Basic Block）识别算法

**定义**：基本块为一段线性指令序列，其入口只有第一个指令，出口只有最后一条（无中间跳入/跳出）。

**识别步骤**：
1. 标记所有可能的基本块入口点（leaders）：
   - 第 0 条指令。
   - 所有跳转目标（跳转 / 条件跳转 / 带返回的 CALL 目标）。
   - 跳转指令后面的指令（fall-through）。
2. 从每个 leader 向后收集指令直到遇到另一 leader 或终止指令（return / tail return / jump）。

**伪代码**：
```
leaders = set([0])
for pc, ins in enumerate(code):
  if ins.isJump():
    leaders.add(ins.target)
    leaders.add(pc+1)
  if ins.isReturn():
    leaders.add(pc+1)

for leader in sorted(leaders):
  block = collect instructions from leader until next leader
  add block to BBList
```

### 3.2 控制流图（CFG）构建技术

**节点**：基本块。
**边**：由跳转、条件分支、异常/调用返回形成的控制转移。

**构建方法**：
- 遍历每个基本块找最后一条指令，依据指令类型建立边：
  - 无条件跳转：单向边到目标块。
  - 条件跳转：边到目标块和 fall-through 块。
  - 返回：无后继。
- 对函数调用，不将其视为基本块间静态跳转（调用返回仍属同一块后继）。

**许多分析（如循环检测、结构化转换）都基于 CFG）。**

### 3.3 分支结构（if-else、switch）识别与还原

**if-else 识别**：
- 识别出形如：`if cond goto L1; ... goto L2; L1: ...; L2:` 的模式。
- 模式匹配算法识别条件跳转 + fall-through/跳转配对。

**switch（表驱动分支）识别**：
- Lua 源 `switch` 通常被编译为跳表或一系列比较+跳转。检测大量相近的比较与跳转链，或者检测索引到跳转目标表（jump table）结构。
- 生成 `switch` 或 `if-elseif` 链取决于密度/可读性。

**策略**：优先恢复结构化控制流：将原始跳转翻译为 `if / elseif / else` 或 `switch`，保持原语义。

### 3.4 循环结构（for、while、repeat）识别与还原

**常见模式**：
- `for`（数值 for）: 初始化 -> 条件检查 -> 体 -> 步进 -> 跳回到条件。
- `while`：条件检查在前，若为真进入体后跳回条件。
- `repeat`：体先执行一次，条件在结尾判定跳回。

**检测方法**：
- 利用 CFG 的强连通分量（SCC）与 back-edges 检测循环头（loop header）和循环体边界。
- 分析循环入口处是否有初始化与步进模式（寄存器递增/递减）以识别为 `for`。

**伪代码（循环头检测）**：
```
compute dominator tree of CFG
for each edge u->v:
  if v dominates u: // back-edge
    mark loop with header v and body = nodes reachable from v without leaving v..u region
```

### 3.5 异常处理结构（try-catch）解析技术

Lua 自身没有内建 `try-catch`，但 `pcall/xpcall` 或 C-side 的异常会被编译为特定控制流模式。若字节码中包含 `SETLIST/TAILCALL` 等异常边界，或存在特殊 error handler 原型，需要：
- 识别 `pcall` 模式（例如调用前压入函数、调用后检测返回值）。
- 若存在 `luaD_throw`/`lua_resume` 等外部调用符号（在嵌入字节码场景），则通过模式检测恢复异常处理语义并生成 `pcall`/`xpcall` 或注释提示。

## 4. 代码生成技术

将分析得到的高层控制流、变量信息、表达式重建结果生成可读的 Lua 源代码。

### 4.1 中间表示（IR）设计与实现

**目标**：设计一个既能保持原始指令信息，又便于高层语法生成的 IR。建议采用三地址式或 SSA 风格 IR。

**推荐结构（示例）**：
```text
FunctionIR {
  params: [name]
  locals: [name]
  blocks: [BasicBlockIR]
}
BasicBlockIR {
  id: int
  instrs: [IRInstr]
  succ: [block_id]
}
IRInstr types: Assign(dst, expr), Call(dsts, func, args), Jump(target), CondJump(cond, true_target, false_target), Return(exprs)
Expr: Var(name) | Const(value) | BinOp(op, left, right) | UnOp(op, operand) | TableCons(entries)
```

**SSA vs 非 SSA**：SSA 有利于表达式重构与优化（常量传播、死代码消除），但生成回到 Lua 源时可能复杂。可以先在 IR 中使用 SSA 做分析优化，随后进行 `destruction of SSA` 回到可命名变量形式。

### 4.2 从指令序列到 Lua 语句的转换规则

**映射原则**：保留语义优先，尽可能恢复原始源码结构（如 `local`、控制结构）。

**常见指令映射示例（Lua 5.1）**：
- `LOADK A Bx` → `R[A] = constants[Bx]` → 在 IR 上转为 `Assign(varA, Const)` 并在后续根据 `locvars` 改名为 `local x = <const>`。
- `MOVE A B` → `R[A] = R[B]` → 可能是局部变量赋值。
- `GETUPVAL/SETUPVAL` → upvalue 相关语义，生成 `env`/`upval` 表示。
- `CALL A B C` → `Call` IR, 将返回值映射回寄存器或局部变量。
- `RETURN A B` → `Return` IR。

**步骤**：
1. 将每条字节码转换为 IRInstr，保留 pc 与原始参数（便于调试）。
2. 在 IR 层运行若干重构 pass：合并连续赋值、消除中间临时、合并表达式。
3. 基于 CFG 结构拼接语句块并生成结构化语法（if/while/for/repeat）。

### 4.3 表达式重建算法

表达式重建为一项核心技术。通常字节码通过多条指令构成复杂表达式，需要逆向合并。

**方法**：
- **短期依赖分析**：寻找一系列 `LOADK`/`MOVE`/`BINOP` 指令最后将结果放入某寄存器且该寄存器未被再次写入之前即被使用；将其折叠为单个表达式节点。
- **数据流（Dataflow）追踪**：对每个变量/寄存器执行 reaching-defs 分析以确定表达式的完整构建链。
- **优先级与括号插入**：根据操作符优先级生成合适括号。

**伪代码（表达式折叠）**：
```
function rebuildExpression(target_reg, pc):
  def_pc = reachingDef(target_reg, pc)
  ins = code[def_pc]
  if ins.op == LOADK: return Const(ins.Bx)
  if ins.op == BINOP:
    left = rebuildExpression(ins.A_left, def_pc)
    right = rebuildExpression(ins.A_right, def_pc)
    return BinOp(ins.opcode, left, right)
  if ins.op == CALL:
    args = [rebuildExpression(arg, def_pc) for arg in ins.args]
    return CallExpr(func, args)
end
```

**注意**：需要限制递归深度并处理循环/相互依赖的情况（通过生成临时变量以保证语义）。

### 4.4 函数定义与调用还原技术

**闭包与 upvalue 对应**：当识别到 `CLOSURE` 指令，生成 `function (...) ... end` 语法，并根据被引用的 upvalue 列出外部依赖（如果必要，使用 `local` 绑定）。

**参数与 `vararg`**：根据 `num_params` 与 `is_vararg` 生成 `function(a,b,...)`。

**边界条件**：
- 嵌套函数：保持嵌套结构与上值捕获语义。
- tail call：若检测到 `TAILCALL`，在生成代码时恢复 `return func(...)`。

### 4.5 代码格式化与可读性优化方法

**目标**：输出既能被 Lua 解释器正确解析，又对人类开发者友好。

**方法**：
- 变量重命名：基于 `locvars` 名称优先，否则按使用语义生成 `idx`/`iter`/`i`/`tbl` 等可读名。
- 常量抽象：将重复复杂表达式抽出为临时局部变量并注释原常量来源。
- 控制结构重写：将低层跳转重写为 high-level 结构（if/else/while/for/repeat）并消除 goto（除非无法结构化时保留 goto 并注释）。
- 代码美化：统一缩进（2 或 4 空格）、空行、注释插入（如 `-- recovered from pc XX`）。

## 5. 高级反编译技术

### 5.1 反混淆（常量加密、控制流平坦化、虚假控制流）识别与还原

**常见混淆方法**：
- **常量加密**：将常量以加密或异或存储，运行时通过解密函数（在字节码中）获得真实值。
- **控制流平坦化**：将原先嵌套结构转换为统一循环 + switch + 状态机，增加不可达/虚假分支。
- **垃圾/虚假代码插入**：插入不影响语义的无用指令或分支。

**识别策略**：
- 常量解密：识别集中式解密函数（通常是一个小函数对常量索引/偏移做运算），尝试符号执行或部分执行以恢复明文常量。若函数简单，可在解析阶段直接模拟其行为（一种「模拟执行引擎」）。
- 控制流平坦化：检测大型 switch-on-state 模式：一个状态寄存器 + dispatch 表。恢复方法：
  - 跟踪状态寄存器的更新与分支，重建原始控制流语义（将状态机转换回条件 / 循环）。
  - 应用模式识别（例如识别 `while true; switch(state) ... end` 模式）。
- 虚假分支过滤：通过静态可达性分析与符号约束（条件是否恒真/恒假）消除不可达或恒假分支。

**实施步骤**：
1. 标记并抽象出可疑解密函数。
2. 实现一个轻量级字节码级模拟器以尝试运行这些解密函数（仅对常量输入）。
3. 应用符号执行/约简将常量替换回明文；更新 constants 表。
4. 重新运行控制流重构（有时解密常量会改变分支目标）。

### 5.2 反调试保护的绕过方法

**保护方式**：检查调试环境（如 `debug` 库），反调试函数，或依赖时间/行为差异的检测。

**绕过策略**：
- 在静态分析中识别并忽略检测语句（例如检测 `debug.getinfo` 的调用），将其当作无副作用的函数来跳过或模拟返回合理值。
- 在模拟器中对时间/环境敏感调用返回硬编码正常值。

**注意**：若某些行为为 C 侧强制（嵌入环境），可能无法完全静态恢复；在文档/注释中指出限制并提供动态分析建议（例如在 sandboxed VM 中运行样本以获取运行时常量）。

### 5.3 字节码优化痕迹的识别与处理

编译器优化（如常量折叠、寄存器分配优化、尾调用消除）会使反编译更难。

**识别**：
- 常量折叠：多个表达式合并为单常量（通过常量池）——检查常量表的异常模式。
- 寄存器重用：变量被重用到相同寄存器位置——通过 `locvars` 生存期 & reaching-defs 恢复独立变量语义。

**处理**：在 IR 中插入转换 pass，拆分被重用的寄存器为独立变量名以提升可读性。

### 5.4 复杂数据结构的重建技术

目标：重建表（table）结构、元方法（metatable）、函数表等。

**方法**：
- 常量表重建：当 table 常量以多条 `SETTABLE` 指令构建，追踪构建序列将其合并成内联表构造器 `{k1=v1, k2=v2}`。
- 元表识别：寻找 `setmetatable` 或 `__index`/`__newindex` 模式（或它们的字节码对应），还原元表定义与绑定位置。
- 对象/类式结构：识别 `:new`/`__index` 的组合，恢复为 `Class = {}` 与 `function Class:new() ... end` 的模式（以提高可读性）。

## 6. 验证与测试方法

### 6.1 反编译代码正确性验证流程

目标：确保反编译的 Lua 源能被 Lua 解释器编译并在语义上与原字节码等价。

**步骤**：
1. **语法检查**：生成代码应通过 `luac -p`（语法检查）或 `luacheck`。
2. **再编译（round-trip）**：将生成源码用匹配版本的 `luac` 编译为字节码，并比较与原始 `.luac`：
   - 字节级相等很难达成（因为编译器 metadata/常量顺序/行号可能不同），但应比较在行为上等价：使用测试用例或函数输入输出比较。
3. **动态行为对比**：
   - 在沙箱 VM 中执行原始字节码与反编译后重新编译的字节码/源码，对比关键函数的输出与副作用（输入数据驱动测试）。
4. **差异归因分析**：自动化生成差异报告：不匹配的函数/常量/控制分支。

**注意**：若字节码包含随机化/环境关联逻辑，需在受控环境下运行测试。

### 6.2 字节码与反编译代码一致性测试方法

- **单位测试**：针对每个 Proto 的公共函数（如 `parseProto`, `decodeInstruction`）编写单元测试。
- **回归测试集**：维护不同 Lua 版本、不同代码风格、混淆样例的字节码样本库。
- **性能测试**：针对大型字节码（多千条指令，复杂嵌套）进行反编译时间与内存基准测试。

### 6.3 异常情况处理与边界测试

- **不完整/损坏字节码**：实现容错解析模式（尽量定位并跳过损坏段，或报告精确错误位置）。
- **未知 opcode/扩展**：当遇到未知 opcode，记录并在输出中插入 `-- TODO: unknown opcode 0xXX` 注释，允许继续解析其余部分。
- **巨型常量/递归深度**：在解析与重建中限制递归深度并使用流式/迭代处理以防 OOM。

### 6.4 性能优化与效率提升策略

- **懒解析**：仅解析需要的部分（比如先解码 code[] 再逐个 Proto 延迟解析 constants/protos）。
- **缓存**：对常用字符串/常量建立缓存，避免重复解码。
- **并行化**：对独立的嵌套 Proto 使用线程池并行解析（避免 I/O 瓶颈）。
- **增量 IR 优化**：在表达式重建阶段采用增量算法以减少重复分析。

### C. 基本块识别与 CFG 构建（伪代码）
```python
def build_basic_blocks(code):
    leaders = set([0])
    for pc, ins in enumerate(code):
        if ins.is_jump():
            leaders.add(ins.target)
            leaders.add(pc+1)
        if ins.is_return():
            leaders.add(pc+1)
    leaders = sorted([l for l in leaders if l < len(code)])
    blocks = []
    for i, leader in enumerate(leaders):
        end = leaders[i+1] if i+1 < len(leaders) else len(code)
        block = BasicBlock(i, leader, end)
        block.instrs = code[leader:end]
        blocks.append(block)
    # build edges
    for block in blocks:
        last = block.instrs[-1]
        if last.is_uncond_jump():
            block.succ.add(find_block_by_pc(last.target))
        elif last.is_cond_jump():
            block.succ.add(find_block_by_pc(last.target))
            block.succ.add(find_block_by_pc(block.end_pc)) # fall-through
        elif last.is_return():
            pass
        else:
            block.succ.add(find_block_by_pc(block.end_pc))
    return blocks
```

```python
def rebuild_expr(reg, pc, reaching_defs):
    def_pc = reaching_defs[(reg, pc)]
    ins = code[def_pc]
    if ins.op == 'LOADK':
        return Const(constants[ins.Bx])
    if ins.op in BINOP_OPCODES:
        lreg, rreg = ins.B, ins.C
        l = rebuild_expr(lreg, def_pc, reaching_defs)
        r = rebuild_expr(rreg, def_pc, reaching_defs)
        return BinOp(opcode_to_op(ins.op), l, r)
    if ins.op == 'CALL':
        func = rebuild_expr(ins.A, def_pc, reaching_defs)
        args = [rebuild_expr(a, def_pc, reaching_defs) for a in ins.args]
        return CallExpr(func, args)
    # fallback: treat as variable
    return Var('v{}'.format(reg))
```

### LuaReader.java（简易字节读取器，封装 endianness 与基础读取操作）
```java
package com.example.luadec;

import java.io.DataInputStream;
import java.io.EOFException;
import java.io.IOException;
import java.io.InputStream;
import java.nio.ByteBuffer;
import java.nio.ByteOrder;

/**
 * 负责根据头部信息读取后续字段：整数、size_t、lua_Number、字符串、Instruction
 * 该类示例实现 little-endian 与 big-endian 两种读取方式（通过 ByteOrder 控制）
 */
public class LuaReader {
    private final DataInputStream in;
    private final ByteOrder byteOrder;
    private final int sizeInt;      // size of int in bytes (通常 4)
    private final int sizeSizeT;    // size of size_t
    private final int sizeInstruction; // e.g. 4
    private final int sizeLuaNumber;   // e.g. 8
    private final boolean integralFlag; // lua_Number 是否为整型

    public LuaReader(InputStream input,
                     ByteOrder byteOrder,
                     int sizeInt,
                     int sizeSizeT,
                     int sizeInstruction,
                     int sizeLuaNumber,
                     boolean integralFlag) {
        this.in = new DataInputStream(input);
        this.byteOrder = byteOrder;
        this.sizeInt = sizeInt;
        this.sizeSizeT = sizeSizeT;
        this.sizeInstruction = sizeInstruction;
        this.sizeLuaNumber = sizeLuaNumber;
        this.integralFlag = integralFlag;
    }

    public int readInt() throws IOException {
        byte[] buf = new byte[sizeInt];
        in.readFully(buf);
        return toInt(buf);
    }

    public long readSizeT() throws IOException {
        byte[] buf = new byte[sizeSizeT];
        in.readFully(buf);
        return toLong(buf);
    }

    public int readInstruction() throws IOException {
        byte[] buf = new byte[sizeInstruction];
        in.readFully(buf);
        return toInt(buf);
    }

    public double readLuaNumber() throws IOException {
        byte[] buf = new byte[sizeLuaNumber];
        in.readFully(buf);
        if (integralFlag && sizeLuaNumber <= 8) {
            // 处理被当作整数的 lua_Number
            long v = toLong(buf);
            return (double) v;
        }
        ByteBuffer bb = ByteBuffer.wrap(buf).order(byteOrder);
        if (sizeLuaNumber == 8) {
            return bb.getDouble();
        } else if (sizeLuaNumber == 4) {
            return bb.getFloat();
        } else {
            // 不常见的宽度
            throw new IOException("Unsupported lua_Number size: " + sizeLuaNumber);
        }
    }

    public String readLuaString() throws IOException {
        long size = readSizeT();
        if (size == 0) return null; // 空串或 null
        int strlen = (int) (size - 1); // 末尾的
        byte[] buf = new byte[strlen];
        in.readFully(buf);
        byte term = in.readByte(); // 读取终止符 ''
        return new String(buf, "UTF-8");
    }

    private int toInt(byte[] buf) {
        ByteBuffer bb = ByteBuffer.wrap(buf).order(byteOrder);
        if (buf.length == 4) return bb.getInt();
        if (buf.length == 2) return bb.getShort();
        if (buf.length == 1) return buf[0];
        // 对于非标准宽度，我们仍支持最长 8 字节并截断
        if (buf.length == 8) return (int) bb.getLong();
        return 0;
    }

    private long toLong(byte[] buf) {
        ByteBuffer bb = ByteBuffer.wrap(padTo8(buf)).order(byteOrder);
        return bb.getLong();
    }

    private byte[] padTo8(byte[] src) {
        byte[] dest = new byte[8];
        int off = 8 - src.length;
        System.arraycopy(src, 0, dest, off, src.length);
        return dest;
    }
}
```

```java
        this.endPc = endPc;
    }

    public int getId() { return id; }
    public int getStartPc() { return startPc; }
    public int getEndPc() { return endPc; }
    public List<Instruction> getInstrs() { return instrs; }
    public void addInstr(Instruction ins) { instrs.add(ins); }

    @Override
    public String toString() {
        return String.format("BB%d[%d:%d]", id, startPc, endPc);
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        BasicBlock that = (BasicBlock) o;
        return id == that.id && startPc == that.startPc && endPc == that.endPc;
    }

    @Override
    public int hashCode() {
        return Objects.hash(id, startPc, endPc);
    }
}
```

### CFGBuilder.java
```java
package com.example.luadec.cfg;

import com.example.luadec.Instruction;
import org.jgrapht.Graph;
import org.jgrapht.graph.DefaultDirectedGraph;
import org.jgrapht.graph.DefaultEdge;
import org.jgrapht.nio.dot.DOTExporter;
import org.jgrapht.nio.Attribute;
import org.jgrapht.nio.DefaultAttribute;

import java.io.StringWriter;
import java.util.*;
import java.util.stream.Collectors;

/**
 * 基本块识别与 CFG 构建
 */
public class CFGBuilder {

    /**
     * 根据指令序列识别基本块（leaders 算法）并返回块列表（按 startPc 排序）
     */
    public static List<BasicBlock> buildBasicBlocks(List<Instruction> code) {
        int n = code.size();
        Set<Integer> leaders = new HashSet<>();
        leaders.add(0);

        for (int pc = 0; pc < n; pc++) {
            Instruction ins = code.get(pc);
            if (ins == null) continue;
            if (ins.isJump()) {
                int target = computeJumpTarget(ins, pc);
                if (target >= 0 && target < n) leaders.add(target);
                if (pc + 1 < n) leaders.add(pc + 1);
            }
            if (isReturnOpcode(ins.getOpcode())) {
                if (pc + 1 < n) leaders.add(pc + 1);
            }
        }

        List<Integer> sortedLeaders = leaders.stream().filter(i -> i >= 0 && i < n).sorted().collect(Collectors.toList());
        List<BasicBlock> blocks = new ArrayList<>();
        for (int i = 0; i < sortedLeaders.size(); i++) {
            int start = sortedLeaders.get(i);
            int end = (i + 1 < sortedLeaders.size()) ? sortedLeaders.get(i + 1) : n;
            BasicBlock bb = new BasicBlock(i, start, end);
            for (int pc = start; pc < end; pc++) bb.addInstr(code.get(pc));
            blocks.add(bb);
        }
        return blocks;
    }

    /**
     * 构建 CFG，返回 JGraphT Graph 对象（顶点为 BasicBlock，边为 DefaultEdge）
     */
    public static Graph<BasicBlock, DefaultEdge> buildCFG(List<Instruction> code, List<BasicBlock> blocks) {
        DefaultDirectedGraph<BasicBlock, DefaultEdge> g = new DefaultDirectedGraph<>(DefaultEdge.class);
        for (BasicBlock b : blocks) g.addVertex(b);

        Map<Integer, BasicBlock> pcToBlock = new HashMap<>();
        for (BasicBlock b : blocks) pcToBlock.put(b.getStartPc(), b);

        for (BasicBlock b : blocks) {
            Instruction last = b.getInstrs().isEmpty() ? null : b.getInstrs().get(b.getInstrs().size() - 1);
            if (last == null) continue;
            int pc = last.getPc();

            if (last.isUnconditionalJump()) {
                int target = computeJumpTarget(last, pc);
                BasicBlock tgt = findBlockForPc(pcToBlock, target);
                if (tgt != null) g.addEdge(b, tgt);
            } else if (last.isConditionalJump()) {
                int target = computeJumpTarget(last, pc);
                BasicBlock tgt = findBlockForPc(pcToBlock, target);
                BasicBlock fall = findBlockForPc(pcToBlock, b.getEndPc());
                if (tgt != null) g.addEdge(b, tgt);
                if (fall != null) g.addEdge(b, fall);
            } else if (isReturnOpcode(last.getOpcode())) {
                // no successor
            } else {
                // fall-through
                BasicBlock fall = findBlockForPc(pcToBlock, b.getEndPc());
                if (fall != null) g.addEdge(b, fall);
            }
        }

        return g;
    }

    private static BasicBlock findBlockForPc(Map<Integer, BasicBlock> pcToBlock, int pc) {
        if (pcToBlock.containsKey(pc)) return pcToBlock.get(pc);
        // 如果 pc 不是某个 block 的 startPc，find the block whose range covers pc
        for (BasicBlock b : pcToBlock.values()) {
            if (pc >= b.getStartPc() && pc < b.getEndPc()) return b;
        }
        return null;
    }

    private static int computeJumpTarget(Instruction ins, int pc) {
        // 根据不同 opcode 类型计算目标：这里示例用 sBx 或 Bx
        // 真实实现需用 OpcodeProperties 提供元信息
        int opcode = ins.getOpcode();
        if (OpcodeProperties.isSbxJump(opcode)) {
            return pc + 1 + ins.getSbx();
        }
        if (OpcodeProperties.isBxJump(opcode)) {
            return ins.getBx();
        }
        // fallback
        return -1;
    }

    private static boolean isReturnOpcode(int opcode) {
        return OpcodeProperties.isReturn(opcode);
    }

    /**
     * 导出 Graph 为 DOT 文本以便可视化
     */
    public static String exportDot(Graph<BasicBlock, DefaultEdge> g) {
        DOTExporter<BasicBlock, DefaultEdge> exporter = new DOTExporter<>(
                v -> String.valueOf(v.getId()),
                BasicBlock::toString,
                e -> "" // edge label
        );
        exporter.setVertexAttributeProvider(v -> {
            Map<String, Attribute> m = new LinkedHashMap<>();
            m.put("label", DefaultAttribute.createAttribute(v.toString()));
            return m;
        });
        StringWriter sw = new StringWriter();
        exporter.exportGraph(g, sw);
        return sw.toString();
    }
}
```

### 说明与可拓展点
1. **OpcodeProperties**：示例中依赖 `OpcodeProperties`（或枚举）来判断某 opcode 是否为 jump/conditional/unconditional/return，以及是否使用 `sBx` 或 `Bx` 作为跳转偏移。你需要为目标 Lua 版本实现该类或枚举，并为每个 opcode 标注元信息。
2. **复杂跳转与异常边**：本实现仅处理常见跳转、条件跳转与返回。对 `FORLOOP`、`TFORLOOP`、`TAILCALL` 等特殊指令，需在 `computeJumpTarget` 中加入对应逻辑并在 CFG 中建立额外边。
3. **性能**：当指令数巨大时，寻找 block 的算法可优化为区间树或有序列表 + 二分查找。
4. **更多分析**：在得到 CFG 后可用 `jgrapht` 的算法模块（如 dominator tree、Tarjan SCC）来检测循环、back-edge、不可达块等，便于后续结构化重建。
